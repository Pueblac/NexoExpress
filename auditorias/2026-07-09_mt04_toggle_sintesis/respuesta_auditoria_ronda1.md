Equivalencia: audioBase64 && !isRecording && duration > 0: condiciones del flujo base preexistentes. !processAudio.isPending: cumple con "guard del efecto ... + !isPending", previniendo ejecuciones mientras muta. **Q2:** Traza de dos grabaciones (A con ON, B con OFF, misma carga):
1. Grabación A finaliza (toggle ON): generarSintesisRef.current = true, isPending = false, lastProcessedAudioRef.current = null. El guard pasa. lastProcessedAudioRef.current → "base64_A". **Payload 1:** { audioBase64: "base64_A", mimeType: "audio/webm", msDuration: duration, generarSintesis: true } (home.tsx:131).
2. Mutación en vuelo y 201: isPending pasa a true y luego a false.

4. Grabación B finaliza: audioBase64 = "base64_B", generarSintesisRef.current = false, isPending = false, lastProcessedAudioRef.current = "base64_A". El guard pasa ("base64_A" !== "base64_B"). **Payload 2:** { audioBase64: "base64_B", mimeType: "audio/webm", msDuration: duration, generarSintesis: false }.

**Q3:** Traza de montaje con preferencia ON y recarga:

2. Promesa resuelve, user se puebla (ej. {uid: "abc"}). ProtectedRoute monta <Home />.
3. <Home /> se renderiza por primera vez. Se ejecuta la función inicializadora de useState<boolean> (home.tsx:55). !user evalúa a false. Ejecuta localStorage.getItem("actaexpress:generarSintesis:abc") === "true". Retorna true. generarSintesis inicializa en true. **El switch inicia en ON.**

**Escenario EXACTO de fallo:** La restauración fallaría exclusivamente si en el futuro se modifica ProtectedRoute de modo que deje de bloquear el renderizado durante la carga, y permitiera montar <Home /> transitoriamente cuando user es null. Si eso ocurre, el inicializador de useState (home.tsx:56) evaluaría if (!user) return false; en el primer montaje, fijando el estado perpetuamente a false para esa sesión, y al poblarse el user un render más tarde, el valor del localStorage nunca volvería a leerse (porque las funciones inicializadoras de useState corren una única vez por montaje).

Según el contexto verificado (CTX-A), el servidor es responsable de leer este booleano, poblar la colección sintesis/{actaId} en background, y purgarlos explícitamente del objeto antes de grabar en la colección actas/. Dado que el backend purga los campos, la base de datos actas/{id} permanece inalterada y libre de contaminación por síntesis.

## 3. Refutación de Trampas Anti-aprobación (T1 - T5)

- **T1 — Regresión B1 encubierta:** NO hay regresión. A diferencia de un !isSuccess defectuoso que se quedaría en false perpetuamente tras el primer éxito bloqueando todo el hook, aquí se utiliza !processAudio.isPending. Al finalizar una grabación exitosa (o fallida), isPending vuelve a false. Al generarse un nuevo audio, audioBase64 es distinto de lastProcessedAudioRef.current, por lo que el guard evaluará limpio (true) y la segunda grabación se procesará sin problemas (home.tsx:123-124).
- **T2 — Flag stale:** El valor enviado siempre es el correcto. Si el componente re-renderea antes de correr el efecto, la línea generarSintesisRef.current = generarSintesis; (home.tsx:60) mantiene actualizada la ref síncronamente en cada fase de render. Cuando el useEffect finalmente evalúa y ejecuta mutate, lee el valor fresco de generarSintesisRef.current (home.tsx:131), el cual reflejará el último click del usuario, sin cierres sobre valores estancos (stale closures).
- **T3 — Fuga entre cuentas:** Es imposible. La clave es dinámica: localStorage.getItem(`actaexpress:generarSintesis:${user.uid}`) (home.tsx:57). Si B entra sin preferencia previa, getItem retorna null, y null === "true" arroja false, otorgándole el default seguro y no la preferencia guardada bajo el UID del usuario A.
- **T4 — Doble POST:** NO hay doble post. Si onSuccess dispara invalidateQueries y fuerza un re-render mientras el estado del audioBase64 sigue poblado, el useEffect correrá de nuevo. Sin embargo, en el disparo original (ANTES de mutar), la instrucción lastProcessedAudioRef.current = audioBase64; (home.tsx:125) ya actualizó la ref. En el segundo paso, la comprobación lastProcessedAudioRef.current !== audioBase64 será falsa, bloqueando la ejecución y previniendo el bucle.
- **T5 — Drift colado:** NO existe drift malicioso. Todo el código en generated/ corresponde punto por punto a los campos descritos en el contexto MT-04a: plataforma, updatedAt, el tipo ActaPlataforma (api-zod/src/generated/types/acta.ts:24-27), y el campo opcional generarSintesis en audioInput (api-zod/src/generated/types/audioInput.ts:13-14). No se observan en el diff modificaciones furtivas en URLs de endpoints, lógicas extra, u otros parámetros.

## 4. Opinión Fundada sobre Dudas (D1 - D3)

- **D1 (id="switch-sintesis" vs data-testid):** No introduce riesgo alguno y es, de hecho, una excelente práctica en React/HTML. El componente <Label htmlFor="switch-sintesis"> requiere forzosamente un id homólogo en el <Switch id="switch-sintesis"> (home.tsx:226) para que el usuario pueda interactuar clickeando el texto. El data-testid sirve propósitos puramente de testing E2E; ambos coexisten sin colisionar.
- **D2 (acoplamiento de persistencia y CTX-B):** Tal como se trazó en Q3, funciona de manera robusta gracias a CTX-B. Considero que no amerita añadir defensas lógicas en esta tanda (p. ej. mover la carga a un useEffect sobre el user) porque incrementaría la complejidad y causaría un "flicker" o "flash" (de false a true) visible para el usuario si hay renders en el medio. Basta documentar fuertemente que Home asume existencia de sesión.
- **D3 (resiliencia del parseo === "true"):** La decisión de evitar JSON.parse en favor de conversión directa a string y posterior comparación estricta === "true" (home.tsx:57) es extremadamente segura. Cualquier valor corrupto (undefined, null, "[object Object]", u otros strings generados por bugs antiguos) fallará la aserción y degradará automáticamente a false (el default seguro), sin arrojar excepciones fatales de parseo que rompan la carga de la UI.
