# Diseño propuesto (claims a verificar)

- **C1 — Regeneración como micro-tarea previa (MT-04a).** Se ejecuta `pnpm --filter @workspace/api-spec codegen`; PROHIBIDO editar a mano los `generated/` (cabecera orval). El diff resultante se revisa y solo se acepta lo derivado del yaml vigente (CTX-2); cualquier otro cambio detiene la MT y se reporta.
- **C2 — Estado y persistencia.** Estado React `generarSintesis` inicializado desde `localStorage` con clave **por usuario** `actaexpress:generarSintesis:{uid}` (evita filtrar la preferencia entre cuentas del mismo navegador); se escribe al cambiar el switch; default `false` si no hay clave.
- **C3 — Envío sin closure stale.** El `useEffect` de CTX-1 tiene las deps congeladas (`eslint-disable`). El valor del toggle se lee vía `useRef` sincronizado (`ref.current = generarSintesis` en cada render) y el mutate incluye `generarSintesis: ref.current` — el flag NO entra a las deps (no debe re-disparar el efecto) y aún así siempre viaja el valor vigente al momento de parar.
- **C4 — UI del toggle.** `Switch` + `Label` de shadcn bajo el botón (zona de CTX-4), `data-testid="switch-sintesis"`, `disabled={isRecording || processAudio.isPending}`, sin layout shift (espacio reservado). Texto: "Análisis profundo" / subtítulo "Transcripción completa y análisis (tarda más)".
- **C5 — Feedback asíncrono honesto.** Si el request llevó `true`, el toast de éxito añade: "El análisis profundo se está generando en segundo plano." SIN polling ni espera de `sintesis/` en esta fase — el 201 del acta sigue mandando y la feature no puede degradar el flujo principal.
- **C6 — Alcance cerrado.** Cero cambios en backend, `openapi.yaml`, schema Firestore, reglas o Android. Solo: regeneración (MT-04a) + `home.tsx` y componentes UI existentes (MT-04b).

# Dudas declaradas por el Arquitecto

- **D1:** ¿`localStorage` por uid es el nivel de persistencia correcto para esta fase, o debería vivir en `users/{uid}` de Firestore (sincroniza entre dispositivos pero exige ciclo de schema)? Propongo localStorage ahora y migración opcional después.
- **D2:** ¿Enviar `generarSintesis` como boolean explícito SIEMPRE (true y false), o incluirlo solo cuando es `true`? El default del server es `false`; el boolean explícito documenta la intención en logs, pero agranda mínimamente el contrato efectivo.
- **D3:** La regeneración (C1) puede traer diffs colaterales (versión de orval, cambios acumulados del yaml no relacionados). Riesgo de scope creep dentro de MT-04a — ¿la salvaguarda "revisar el diff y detenerse si hay cambios ajenos" es suficiente?
- **D4:** El toggle bloqueado durante la grabación (CTX-5) obliga a decidir ANTES de grabar, pero el flag se consume al PARAR: técnicamente podría permitirse cambiarlo en caliente hasta el stop. ¿El bloqueo aprobado crea algún caso de frustración o inconsistencia que amerite reabrirlo con el Director?
